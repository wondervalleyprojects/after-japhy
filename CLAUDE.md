## ARCHITECTURE: LAZYGRAPHRAG

LazyGraphRAG defers graph construction until query time. No upfront entity extraction across 75k records. No pre-built knowledge graph. No embedding costs.

### Index Build (one-time, cheap — zero LLM calls)

1. Load `master_corpus_anon.jsonl`
2. Chunk each record's content into passages of ~400 words with 80-word overlap
3. Store chunks in SQLite with schema: `(chunk_id TEXT, record_id TEXT, source_name TEXT, date TEXT, content TEXT, word_count INTEGER)`
4. Enable FTS5 virtual table over content for fast full-text search
5. Build BM25 index using `rank_bm25.BM25Okapi` over all chunk texts
6. Serialize BM25 index to `bm25_index.pkl`
7. Store chunk ID → chunk text mapping in `chunks_lookup.pkl`

Target: ~200,000–300,000 chunks from 75,816 records.

### Query Time (per conversation turn — 2 Haiku calls)

```
Turn arrives
  ↓
Step 1 — BM25 Retrieval
  Query → BM25 search → top 20 chunks
  Also run SQLite FTS5 search → top 10 chunks
  Merge, deduplicate, keep top 15 by combined score
  ↓
Step 2 — Entity Extraction (Haiku call #1)
  Pass top 15 chunks to Haiku with this instruction:
  "Extract entities and relationships from these passages.
   Return JSON: {entities: [{name, type, weight}], relationships: [{from, to, label, strength}]}
   Entity types: THEME, PERSON, PLACE, PERIOD, EMOTION, BELIEF, PROJECT.
   Weight = frequency × contextual significance (1–5 scale).
   Relationships: co-occurrence, contradiction, evolution, cause, recurrence."
  ↓
Step 3 — Mini-Graph Construction
  Build NetworkX DiGraph from extracted entities + relationships
  Nodes = entities (weighted by extraction weight)
  Edges = relationships (weighted by strength)
  Compute top 5 nodes by weighted degree centrality
  Format graph summary as plain text:
  "Thematic web: [node] ←[relation]→ [node] ..."
  ↓
Step 4 — Synthesis (Haiku call #2)
  System prompt: system_prompt.txt (The Incomplete Reader persona)
  Context injected before conversation history:
    CORPUS PASSAGES:
    [source_name, date] text
    ... (top 15 chunks)
    
    THEMATIC WEB (from this record):
    [graph summary]
  User turn: visitor's message
  Response: The Incomplete Reader speaks
```

### Cost profile

- Index build: $0 (no LLM calls)
- Per conversation turn: ~2,000 tokens input + ~400 tokens output to Haiku
- Haiku pricing: ~$0.0003 per turn
- Opening reflection: 1 additional Haiku call at session start

### Phase 2 — Victorian Name Anonymizer
Script: `scripts/anonymize_corpus.py`
Input: `master_corpus.jsonl`
Output: `master_corpus_anon.jsonl` (in same directory as input)

**Rules:**
- Replace private personal names (first name, last name, or both) with `[Initial]—`
  - Example: "Spencer told me" → "[S—] told me"
  - Example: "I called David back" → "I called [D—] back"
  - Full names: "Michael Torres" → "[M—]"
- **Preserve:** place names, brand names, company names, famous/public figures (celebrities, politicians, historical figures, authors, artists, musicians)
  - Lady Gaga stays "Lady Gaga"
  - Los Angeles stays "Los Angeles"
  - Wonder Valley stays "Wonder Valley"
  - Google stays "Google"
- **Preserve:** "Japhy" and "Japhy Grant" (the subject — not anonymized)
- **Preserve:** names that appear in URLs

**Implementation strategy:**
- Use spaCy `en_core_web_sm` NER to identify PERSON entities
- Cross-reference against a safelist of famous names (build from Wikipedia's "common celebrity first names" or use a simple heuristic: if the name appears in more than 50 records across the corpus, it is likely public-facing)
- For ambiguous cases (names that could be both private and public), prefer anonymization
- Generate a consistent mapping: if "Spencer" → "[S—]" in one record, it must be "[S—]" everywhere
- If same initial appears for multiple names: "Spencer" and "Sarah" both → "[S—]" (intentional — this is the Victorian style, ambiguity is a feature)
- Log the name→initial mapping to `name_mapping_log.json` for review (not committed to git)
- Preserve metadata fields unchanged — only anonymize the `content` field

**Verification:** Sample 50 random records from output, print to console for manual review before proceeding.

## PERSONA: THE INCOMPLETE READER

The system prompt governs the persona. The architecture serves it. Do not introduce any behavior, routing logic, or response formatting that contradicts the persona. Specifically:

- Do not add "helpful" fallback behaviors for when retrieval fails. If retrieval fails, the reader says so honestly: "the record doesn't yield anything on that."
- Do not add conversation starters, suggested prompts, or "try asking about..." UI elements. The reader is not a chatbot assistant.
- Do not expose source metadata as citations in a scholarly format. Sources are shown as a muted line — they are not footnotes.
- The opening reflection must come from actual corpus content (a random high-scoring chunk), not a hardcoded string.
